Design Principles

Posted on 2019-09-22 | In Design Pattern
A summary of six design principles with examples: Single Responsibility Principle, Liskov Substitution Principle, Dependency Inversion Principle, Interface Separation Principle, Dmitry's Principle and Open Close Principle.

This is my first post of design patterns. I will discuss the famous six principles of design patterns in this post hopefully with some examples. And leave the discussions of specific design patterns to other posts. Feel free to check out other posts under the category Design Patterns.

Single Responsibility Principle (SRP)

Key point: A class should always have only one responsibility.

Explanation: Class A is responsible for task 1 and task 2. If class A needs to be changed due to the chance of task 1, task 2 might be affected. Therefore, we should have class A which is only responsible for task 1 and class B for task 2.

SRP seems to be very easy for the first glance and most programmers will follow the principle in their daily work, even though they might not aware of its name. However, even those experienced programmers sometimes can write code that violates this principle, due the increased complexity of the program. For example, a senior software engineer may have a program where class A is responsible for task 1. As the software evolves, task 1 may be subtask 11 and subtask 12. In this case, they might choose to rewrite the class A directly, which is less time-consuming, especially when they are working on a relatively complex system. However, this violates the SRP and might bring unexpected complexity to the system. (What if the task 1 has more subtask or the subtasks are divided into multiple subtasks in the future?) A better way to achieve it is to break class A into two subclasses, probably by making class A a parent class or abstract class, and have class A responsible for subtask 11 and class B for subtask 12. This seems to be very tedious but it definitely worths the effort.

However, this really depends on the real project you are working on. You don’t have to strictly follow the SRP because of the extra work brought by adjusting the existing classes. There is a simple principle that I follow all the time: always have the design pattern in your mind and do code refactoring as early as possible.

Liskov Substitution Principle (LSP)

Key point: If S is a subtype of T, then objects of type T can be replaced with objects of type S without altering any of the desired properties of the program (correctness, task performed, etc).

Subclasses can extend the functionality of their parent classes. But do not alter their parent classes’ existing functions. Overriding parent class’s methods are more dangerous than just adding methods in subclasses. However, this is not without risk. If the parent class acquires a new method and you are very unlucky to have given the same method signature to the subclasses before, the subclass won’t compile.

This is the fundamental of inheritance. Inheritance brings convenience to programmers but also dangers. Inheritance actually violates another OOD design principle which is encapsulation. From the book Effective Java, there are two best practices to work around this:

  • Item 18: Favor composition over inheritance

    A subclass depends on the implementation of its superclass for its proper function. From time to time, the implementation of the superclass will change and breaks the subclass. Therefore, a subclass must evolve in tandem with its superclass, unless the superclass’s authors have designed and documented it specifically for the purpose of being extended.

    Composition: Give your new class a private field that references an instance of the existing class. Each instance method in the new class invokes the corresponding method on the contained instance of the existing class.

    Take a look at an example, we want to create an InstrumentedSet class which can count the addition of elements to the set. We can create a forwarding class which contains all of the forwarding methods and nothing else. And have our main class extends the forwarding class and add instrumentation to it. This is also known as the Decorator Pattern because the InstrumentedSet class “decorates” a set by adding instrumentation.

  • Item 19: Design and document for inheritance or else prohibit it

    A best practice is: never extend a class that is not designed or documented explicitly for inheritance.

    First, the class must document precisely the effects of overriding any method, including which overridable methods the method invokes, in what sequence, and how the results of each invocation affect subsequent processing.

    For example, the remove method in java.util.AbstractCollection package relies on the iterator method. Therefore, in the documentation of remove method, there should be a special section called Implementation Requirements to indicate how it works and the effect of overriding iterator method.

    Constructors must not invoke overridable methods, directly or indirectly. The best solution is to prohibit subclassing in classes that are not designed and documented to be safely subclassed. The first way to achieve this is to make the class a final class. The alternative is to make all the constructors private or package-private and to add public static factories in place of constructors. If inheritance is necessary, try to eliminate the class’s self-use of overridable methods entirely, makes the class reasonably safe to subclass because overriding a method will never affect the behavior of any other method.

Dependency Inversion Principle (DIP)

Key point: Interface-oriented programming

Problem: Class A depends on class B. If there is a time that class A needs to depend on class C instead of class B, we need to modify class A’s code. However, typically class A is responsible for high level logic and class B and C are responsible for low level operations. Directly modify class A will bring unnecessary threat to the program.

Solution: Class A depends on interface I and class B and C implement interface I separately. Class A interacts with class B or C through interface I.

Let’s take an example form my real experience. There is a service that needs to do CRUD operations of a database. However, for future scalability, we have to consider the possibility of changing the underlying database. We can have our PersistentService depends on a DatabaseManager interface. And different database will implement the same DatabaseManager interface, which defines the operations that our PersistentService needs to perform. Any other new database can plugin in this way by implementing the same interface.

If in some time, we need to change our database from Redis to MongoDB, we just need to write a MongoDBManager class which implements the DatabaseManager interface. And PersistentService can safely use the new MongoDBManager without changing its implementation.

Interface Segregation Principle (ISP)

Key point: Divide a large interface into multiple small interfaces, each of which is responsible for only a few related operations.

Class A, B depend on class C, D separately through interface I. Interface I is a large interface with multiple methods. Therefore, class C and D have to implement all methods although not all of them are of interest. make the inheritance structure very clear and easy to maintain.

We can divide interface I into small interfaces I1, I2, I3. Class B implements I2 and I1 with methods 1, 2, 3 and class D implements I1 and I3 with methods 1, 4, 5. This is very flexible and reduce the size of interface which makes it easier to maintain and extend in the future.

Dmitry’s Principle (Least Knowledge Principle)

Key point: An entity should interact with other entities as least as possible. An object should have the least knowledge of other objects.

With Dmitry’s Principle, we can have more independent modules, so that when we modify a module, other modules that depend on it will be affected as least as possible. Classes should encapsulate their information within the class as much as possible and provide public methods for others to access.

If a class jumps more than two times to access another class, you need to think of a way to reconstruct them. The more the number of jumps, the more complex the system and the more difficult to maintain.

For example, CompanyManager interacts with the SubEmployee through SubCompanyManager. However, CompanyManager doesn’t need to know about the SubEmployee. It can achieve getting all the employees by having the SubCompanyManager to get their own employees and sum up in the CompanyManager class.

We can modify the Branch to add a printAllEmployees, so that the HeadQuarter class won’t have to interact with the BranchEmployee directly.

Open Close Principle

Key point: A software entity like class, module and function should be open for extension but close for modification.

Requirements evolve during the life cycle of the software. When there are new requirements, we should try to extend the existing entities instead of modifying them. This principle is the most fundamental principle in OOD, but it is mostly liked a summary of the previous five principles or a score to measure how well designed is the software.

  1. Single Responsibility Principle tells us to design classes with single responsibility.
  2. Liskov Substitution Principle tells us not to modify parent classes’ functions. Instead, try to extend them or use composition over inheritance.
  3. Dependency Inversion Principle brings us the concept of Interface-Oriented Programming.
  4. Interface Separation Principle favors multiple small interfaces over a single large interface.
  5. Dmitry’s Principle teaches us a software design principle: low coupling and high aggregation.
  6. Out of all, the Open Close Principle is a summary of those principles: favor extendability over modification.

These are the six famous design principles of good software. With those design principles in mind, we can design more maintainable and scalable softwares.
