How to Write a good CL

发表于 2019-09-25 | 分类于 Code Review | | 阅读次数:
This post is a summary of The Google's Engineering Practices documentation about how to write a good CL.

Writing good CL description

A CL description is a public record of what change is being made and why it was made. A lot of developers do very well in writing what the CL does but tend to ignore why it was made.

First Line

  • Short summary of what is being done.
  • Complete sentence, written as it was an order.
  • Follow by empty line.

The first line should be a short summary of what is being made. It should be informative enough so that future developers don’t have to read your CL or its whole description just to get a general idea of what your CL actually did.

Use an imperative sentence as if you are making an order. For example:

  • Delete the FizzBuzz RPC and replace it with the new system.

Instead of:

  • Deleting/Deleted the FizzBuzz RPC and replacing/replaced it with the new system.

However, you don’t have to use imperative sentences in the rest of the description.

Body

The body of a CL should be informative, including a brief description of the problem that’s being solved, and why it is the best approach. If there are any shortcomings to the approach, they should be mentioned. If relevant, include background information such as bug numbers, benchmark results, and links to design documents.

Even small CLs deserve a little attention to detail. Put the CL in context.

Bad CL Descriptions

  • “Fix build.”
  • “Add patch.”
  • “Moving code from A to B.”
  • “Phase 1.”
  • “Add convenience functions.”
  • “kill weird URLs.”

Follow the above instructions on how to write a good description. Keep in mind that the description is to help future users (and possibly the future YOU) to know what was happening by just looking into the it.

Good CL Descriptions

Here are some examples of good descriptions.

Functionality change

rpc: remove size limit on RPC server message freelist.

Servers like FizzBuzz have very large messages and would benefit from reuse. Make the freelist larger, and add a goroutine that frees the freelist entries slowly over time, so that idle servers eventually release all freelist entries.

The first few words describe what the CL actually does. The rest of the description talks about the problem being solved, why this is a good solution, and a bit more information about the specific implementation.

Refactoring

Construct a Task with a TimeKeeper to use its TimeStr and Now methods.

Add a Now method to Task, so the borglet() getter method can be removed (which was only used by OOMCandidate to call borglet’s Now method). This replaces the methods on Borglet that delegate to a TimeKeeper.

Allowing Tasks to supply Now is a step toward eliminating the dependency on Borglet. Eventually, collaborators that depend on getting Now from the Task should be changed to use a TimeKeeper directly, but this has been an accommodation to refactoring in small steps.

Continuing the long-range goal of refactoring the Borglet Hierarchy.

The first line describes what the CL does and how this is a change from the past. The rest of the description talks about the specific implementation, the context of the CL, that the solution isn’t ideal, and possible future direction. It also explains why this change is being made.

Small CL that needs some context

Create a Python3 build rule for status.py.

This allows consumers who are already using this as in Python3 to depend on a rule that is next to the original status build rule instead of somewhere in their own tree. It encourages new consumers to use Python3 if they can, instead of Python2, and significantly simplifies some automated build file refactoring tools being worked on currently.

The first sentence describes what’s actually being done. The rest of the description explains why the change is being made and gives the reviewer a lot of context.

Small CLs

Reviewers have discretion to reject your change outright for the sole reason of it being too large.

The right size of a CL is one self-contained change. It makes a minimal change that addresses just one thing. This is usually just one part of a feature, rather than a whole feature at once. The CL should contains everything that the reviewers needs to understand.

There is no hard and fast rules about how large is “too large”. 100 lines is usually a reasonable size for a CL, and 1000 lines is usually too large. The number of files that a change is spread across also affect its “size”. A 200-line change in one file might be okay, but spread across 50 files it would usually be too large.

Unlike the developers who have been intimately involved with their code, the reviewers often don’t have much context. Try to write CLs that are smaller than you think you need to write. Reviewers won’t complain about getting CLs that are too small.

It is best to do refactorings in a separate CL from feature changes or bug fixes. But small changes like renaming variables can still go into the same CL.

Keep related test code in the same CL. But independent test modifications can go into separate CLs first. That includes:

  • validating pre-existing, submitted code with new tests
  • refactoring the test code (eg. introduce helper functions)
  • introducing larger test framework code (eg. an integration test)

Evert CL should not break the build for even one minute. Try to come up a way to make sure the whole system keeps working after the CL is submitted if you have multiple CLs that depend on each other.

If a large CL is necessary (which is extremely rare), talk to your reviewers first to let them know what is coming. And expect a longer code review process.

Handle reviewer comments

Don’t take it personally. Never respond in anger to code review comments.

If a reviewer says that they don’t understand something in your code, clarify the code itself first and add necessary comments in the code. Only when a comment seems pointless should you response an explanation in the code review tool. Remember that the comments in the code will help the future users more than those in the code review tool.

No matter how certain you are when you receive a feedback in the CR, step back and consider if the reviewer is providing valuable feedback that will help the codebase and the company. If you can’t answer that question, try to reach out to the reviewers and have them clarify their comments.
