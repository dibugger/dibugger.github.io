Java Synchronizers

发表于 2018-05-08 | 分类于 Java | | 阅读次数:
Java synchronizers with examples, including CountDownLatch, FutureTask, Semaphores, Barriers and more.

A synchronizer is an object that coordinates the control flow of threads based on it state. All synchronizers share certain structural properties: they encapsulate state that determines whether threads arriving at the synchronizer should be allowed to pass or forced to wait, provide methods to manipulate that state, and provide methods to wait efficiently for the synchronizer to enter the desired state.

CountDownLatch

Delay the progress of the threads until the latch reaches the terminal state. The CountDownLatch has an initial value, we can decrease the value by calling the countDown() method. When the value equals to 0, the latch reaches its terminal state and all threads can pass. The state cannot be changed so the gate remains open.

  • Ensuring that a service does not start until all other services on which it depends have started.
  • Ensuring that a computation does not proceed until resources it needs have been initialized.
  • Waiting until all the parties involved in an activity, for instance the players in a multi-player game, are ready to proceed.

FutureTask

FutureTask is used to execute a long task which can return a result. The task is executed in a separate thread while each other thread should be able to get the task result by calling the get() method. If the task is not yet done, the thread is block on get() until the task transitions to the completed state and then returns the result or throws an exception. FutureTask guarantees that this transfer constitutes a safe publication of the result.

Semaphore

Semaphores are used to control the number of activities that can access certain resource or perform a given action at the same time. Activities can acquire a permit and release it when they are done with them. If no permit is available, acquire blocks until one is available. A binary semaphore can be used as a mutex with non-reentrant locking semantics; whoever holds the sole permit holds the mutex. Semaphores can be used to implement a resource pool such as the database connection pool.

Constructor: Semaphore(), Semaphore(int permit, boolean fair)

Main methods:

  • acquire(): acquire a permit, blocks until one is available, or the thread is interrupted
  • acquire(int permit): acquire the given number of permits, blocks until all are available, or the thread is interrupted
  • release(): release a permit to the pool
  • release(int permit): release the given number of permits to the pool
  • tryAcquire(): return true when there is an available permit and acquire it, return false when there is no available permit unblocking

Barriers

With a barrier, all the threads must come together at a barrier point at the same time in order to proceed. Latches are for waiting for events; barriers are for waiting for other threads. An example of the idea is that five friends are agreed to meet at the Building 23 at CMU SV campus at 3PM. Once all of them arrive, they will see what to do next.

A CycliBarrier allows a fixed number of parties to rendezvous repeatedly at a barrier point. Barriers are often used in simulations, where the work to calculate one step can be done in parallel but all the work associated with a given step must complete before advancing to the next step.
